# Multi-Agent Benchmark Campaign

Read this document only through the multi-agent skill for an explicitly parallel benchmark campaign. Read the base benchmark-campaign, project, development, pull-request, review, [multi-agent review](review.md), and [multi-agent issue campaign](issue-campaign.md) procedures before acting.

The base benchmark-campaign skill owns authorization, launch gates, corpus order, harness operation, requirement-fidelity judgment, evidence attribution, the knowledge base, issue bodies, permanent results ledger, and completion gate. This document overrides only discovery and implementation topology.

## Keep Generations Sequential

Never parallelize canonical benchmark generations. One merged and fully validated repository state produces one attributable run. A confirmed automovie-owned defect gates the next run until its implementation batches and integrated cleanup merge.

Do not create agents to wait on a generation or poll its stream. The lead observes it at genuine decision boundaries and begins parallel discovery only after the run has produced an inspectable snapshot.

## Discover In Parallel

Use [review.md](review.md)'s Parallel Issue Discovery Rounds. Give every reviewer the complete repository surface, current corpus, exact run record, external-client trace, rendered output, and requirement-fidelity matrix.

Every reviewer independently inspects the whole surface and the benchmark record. The lead reopens primary evidence, attributes each candidate to automovie, the harness, or the external model, and combines, splits, rewrites, rejects, or defers it in `.wiki` before publication.

## Implement In Coarse Batches

Use [issue-campaign.md](issue-campaign.md)'s coarse DAG batching, parallel claims, local and CI validation, red-CI repair, worktree cleanup, and integrated formatting procedure. Add benchmark origin, scenario tier, and attribution to every batch ledger.

Prefer the smallest number of maximal cohesive batches. Open an implementation agent only for an immediately executable, non-overlapping batch.

After all batches merge, complete the integrated cleanup, remove every worktree and assignment-owned temporary asset, and return to the base benchmark skill's launch gate.

## Prove Completion

Run a new canonical benchmark generation from the fully merged and validated target. Repeat parallel full-scope discovery against that run and repository state.

The campaign succeeds only when every scenario in the declared corpus has a current canonical run whose engine output and viewer rendering satisfy the expected observable result, every automovie-owned shortfall is resolved, and no meaningful candidate survives lead fact-checking after every reviewer completes the whole surface.

If any gate, requirement, issue, worktree, or external asset remains unresolved, continue the cycle or report the exact blocker.
